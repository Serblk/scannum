КРАТКАЯ ШПАРГАЛКА ПО GIT И GITHUB
=================================

Репозиторий проекта:
https://github.com/Serblk/scannum


1. ЧТО ТАКОЕ GIT И GITHUB
-------------------------

Git — программа, которая хранит историю изменений файлов на компьютере.
GitHub — сайт, на который можно загрузить Git-репозиторий и хранить его удалённую копию.

Основные понятия:

- repository (репозиторий) — папка проекта вместе с историей изменений;
- commit (коммит) — сохранённая точка истории проекта;
- branch (ветка) — отдельная линия разработки;
- main — основная ветка проекта;
- remote — удалённый репозиторий, например GitHub;
- origin — короткое локальное имя удалённого репозитория;
- working tree — текущие файлы в папке проекта;
- staging area — список файлов, подготовленных к следующему коммиту.


2. ПЕРВОНАЧАЛЬНАЯ НАСТРОЙКА GIT
-------------------------------

Эти команды обычно выполняются один раз на компьютере:

git config --global user.name "Ваше имя"
git config --global user.email "ваша-почта@example.com"

Проверка:

git config --get user.name
git config --get user.email

Параметр --global означает, что настройка действует для всех проектов пользователя.


3. СОЗДАНИЕ ЛОКАЛЬНОГО РЕПОЗИТОРИЯ
----------------------------------

Перейти в папку проекта и выполнить:

git init

Команда git init создаёт скрытую папку .git с историей проекта.
В нашем проекте это уже сделано, повторять не нужно.


4. ПЕРЕИМЕНОВАНИЕ ОСНОВНОЙ ВЕТКИ
--------------------------------

git branch -M main

Расшифровка:

- git branch — работа с ветками;
- -M — принудительно переименовать текущую ветку;
- main — новое имя основной ветки.

Важно: правильно писать branch, а не bash.


5. ПРОВЕРКА ФАЙЛОВ
------------------

git status

Команда показывает:

- изменённые файлы;
- новые файлы;
- удалённые файлы;
- файлы, подготовленные к коммиту;
- текущую ветку.

Перед каждым коммитом рекомендуется выполнять git status.


6. ПОДГОТОВКА ФАЙЛОВ К КОММИТУ
------------------------------

Добавить все изменения:

git add .

Точка означает все изменения в текущей папке и её подпапках.

Добавить только один файл:

git add README.md

Команда git add не загружает файлы на GitHub. Она только готовит их к коммиту.


7. СОЗДАНИЕ КОММИТА
-------------------

git commit -m "Краткое описание изменений"

Расшифровка:

- git commit — сохранить подготовленные изменения в локальной истории;
- -m — указать сообщение коммита прямо в команде;
- текст в кавычках — описание выполненной работы.

Примеры хороших сообщений:

git commit -m "Добавить интерфейс администратора"
git commit -m "Исправить восьмичасовой таймер"
git commit -m "Добавить тесты очистки истории"

Коммит пока хранится только на компьютере. Для отправки на GitHub нужен git push.


8. ПОДКЛЮЧЕНИЕ GITHUB
---------------------

git remote add origin https://github.com/Serblk/scannum.git

Расшифровка:

- git remote — работа с удалёнными репозиториями;
- add — добавить новый удалённый репозиторий;
- origin — его короткое имя;
- URL — адрес репозитория на GitHub.

В нашем проекте origin уже подключён. Повторять команду не нужно.

Проверить подключение:

git remote -v


9. ПЕРВАЯ ОТПРАВКА НА GITHUB
----------------------------

git push -u origin main

Расшифровка:

- git push — отправить локальные коммиты на GitHub;
- -u — запомнить связь текущей ветки с удалённой веткой;
- origin — удалённый репозиторий GitHub;
- main — ветка, которую отправляем.

После успешного git push -u origin main Git запоминает связь main → origin/main.
В дальнейшем достаточно писать:

git push


10. ОБЫЧНЫЙ ЕЖЕДНЕВНЫЙ ЦИКЛ
---------------------------

После изменения программы:

1) Посмотреть изменения:

git status

2) Подготовить их:

git add .

3) Создать коммит:

git commit -m "Описание изменений"

4) Отправить на GitHub:

git push

Коротко:

git status
git add .
git commit -m "Описание изменений"
git push


11. ПОЛУЧЕНИЕ ИЗМЕНЕНИЙ С GITHUB
--------------------------------

git pull

Команда скачивает новые коммиты с GitHub и объединяет их с локальной веткой.

Полезно выполнять git pull перед началом работы, если проект изменялся на другом компьютере.

Команда состоит из двух действий:

- git fetch — скачать информацию и коммиты;
- git merge — объединить изменения с текущей веткой.


12. ПРОСМОТР ИСТОРИИ
--------------------

Краткая история:

git log --oneline

История с ветками:

git log --oneline --graph --decorate --all

Показать изменения, которые ещё не добавлены через git add:

git diff

Показать изменения, уже подготовленные к коммиту:

git diff --staged


13. РАБОТА С ВЕТКАМИ
--------------------

Показать ветки:

git branch

Создать новую ветку и перейти в неё:

git switch -c feature-название

Перейти в существующую ветку:

git switch main

Объединить готовую ветку с main:

git switch main
git pull
git merge feature-название
git push


14. ПОЛЕЗНЫЕ БЕЗОПАСНЫЕ КОМАНДЫ
-------------------------------

Убрать файл из подготовленных, не удаляя его изменения:

git restore --staged имя-файла

Посмотреть текущий коммит и ветку:

git status
git branch -vv

Посмотреть удалённый адрес:

git remote -v


15. ФАЙЛ .gitignore
-------------------

.gitignore перечисляет файлы, которые Git не должен загружать.

В нашем проекте исключены:

- .venv — виртуальное окружение Python;
- data/*.db — база SQLite;
- captures — фотографии автомобилей;
- models/cache — загруженные модели распознавания;
- reports — созданные отчёты;
- __pycache__ и *.pyc — временные файлы Python.

Перед git add . всегда полезно выполнить git status и убедиться, что там нет:

- паролей и токенов;
- базы данных;
- фотографий;
- личных документов;
- больших временных файлов.


16. ЕСЛИ GITHUB ПРОСИТ ПАРОЛЬ
-----------------------------

Обычный пароль аккаунта GitHub для Git не используется.
Обычно Windows открывает браузер и выполняет вход через Git Credential Manager.

Никогда не вставляйте токены и пароли в код, config.toml, README или коммиты.


17. ТЕКУЩИЕ ДАННЫЕ НАШЕГО ПРОЕКТА
---------------------------------

Локальная ветка: main
Удалённый репозиторий: origin
Адрес: https://github.com/Serblk/scannum.git
Связь веток: main → origin/main

Для следующего обновления достаточно:

git status
git add .
git commit -m "Описание изменений"
git push
